# 2D 线上办公室（含 Web3 资产化）需求梳理 v0.1

## 0. 背景与目标

基于你提出的方向，我们要构建一个**低耦合、可扩展**的线上办公室系统，优先落地 2D 版本，并把“办公室”和“员工”都设计为可资产化对象（支持 Web3 交易/流转）。

核心目标：

1. 先评估并承接既有代码仓能力。
2. 抽离“2D 办公室”成为独立模块（办公室可切换、可出售）。
3. 抽离“员工”成为独立模块（可是真实员工，也可电子员工）。
4. 明确“办公室引入员工”流程：**办公室可引入员工，但不能直接创建员工**。

## 2. 产品范围（Scope）

### 2.1 本阶段（MVP）包含

- 2D 办公室基础体验：房间、座位、基础互动。
- 办公室模块化：办公室模板可创建、上架、购买、切换。
- 员工模块化：员工独立建模，可被办公室“引入/移除”。
- Web3 资产化基础：
  - 办公室作为可交易资产；
  - 电子员工作为可交易资产；
  - 真实员工仅身份映射，不默认可交易。
- 基础权限与关系：公司/办公室管理员可发起“引入员工”流程。

### 2.2 本阶段不包含（可后续迭代）

- 复杂战斗化/游戏化机制。
- 跨链互操作和多链部署。
- 高级治理（DAO 投票、复杂分红）。
- 复杂 AI 员工行为系统（本阶段只保留接口与扩展位）。

## 3. 关键业务概念定义

### 3.1 办公室（Office）

- 独立模块，定义空间、布局、容量、主题等。
- 可以被公司持有、转让、出售。
- 关注“容器”能力：容纳员工、承载在线状态与互动。

### 3.2 员工（Employee）

- 独立模块，不依赖单一办公室存在。
- 分为两类：
  - 真实员工（HumanEmployee）：映射现实身份，受合规约束；
  - 电子员工（DigitalEmployee）：可链上资产化与交易。
- 生命周期由员工模块管理，不由办公室直接创建。

### 3.3 引入关系（Assignment / Residency）

- 表示“某员工在某时间段入驻某办公室”。
- 由办公室发起，但由员工模块/权限系统校验并写入关系。
- 支持历史记录（用于审计、统计、组织变迁追踪）。

### 3.4 资产化对象（Asset）

- 办公室资产（OfficeAsset）。
- 电子员工资产（DigitalEmployeeAsset）。
- 资产所有权和交易行为与业务身份解耦（钱包地址 ≠ 平台账号，但可绑定映射）。

---

## 4. 架构原则（重点：低耦合）

1. **模块解耦**：Office / Employee / Asset 三个域分离。
2. **单向依赖**：
   - Office 依赖 Assignment 接口，不依赖 Employee 内部实现；
   - Employee 不依赖 Office 渲染逻辑；
   - Asset 通过事件或适配器接入业务。
3. **统一身份映射层**：平台用户、员工实体、钱包地址分层建模。
4. **事件驱动优先**：关键动作（入驻、转让、下架、交易完成）输出领域事件。
5. **可替换渲染层**：2D 作为一个前端子模块，后续可扩展 3D。

## 5. 模块需求拆解

## 5.1 代码基线盘点模块（补做项）

目标：在接入仓库代码后，1~2 天内产出“现状能力地图”。

输出物：

- 目录结构图（前后端、合约、脚本）。
- 已有功能清单（登录、房间、资产、交易等）。
- 技术栈与关键依赖。
- 可复用组件与需重构清单。

验收标准：

- 新成员可在 30 分钟内理解现有系统边界。
- 能明确 MVP 里“直接复用”和“需要新增”的项。

## 5.2 2D 办公室模块（Office Runtime）

功能需求：

- 办公室模板管理：创建模板、配置布局（工位/会议桌/公共区域）。
- 办公室实例化：公司选择模板后生成可使用实例。
- 在线空间基础交互：进入办公室、移动、就座、状态展示（在线/忙碌/离开）。
- 办公室切换：公司可在不丢失员工资产的情况下替换办公室。
- 办公室交易预留：办公室实例/模板可上架市场（与资产模块联动）。

非功能需求：

- 50~100 并发在线（MVP 指标）。
- 客户端断线重连后可恢复位置信息。

## 5.3 员工模块（Employee Domain）

功能需求：

- 员工实体管理：创建/认证员工档案（由员工域或 HR 流程创建）。
- 员工类型管理：真实员工 vs 电子员工。
- 员工状态：可用、在职、冻结、离职（电子员工可交易中锁定）。
- 电子员工交易资格：支持上架、转移、下架、授权。
- 入驻关系 API：
  - `assignEmployeeToOffice`（引入）
  - `removeEmployeeFromOffice`（移出）
  - `listOfficeEmployees`（办公室员工视图）

约束要求：

- 办公室模块不得直接调用“创建员工”能力。
- 引入员工必须经过权限校验（公司权限 + 员工可分配状态）。

## 5.4 Web3 资产模块（Asset + Web3 Adapter）

功能需求：

- 资产标准抽象：办公室资产、电子员工资产统一接口。
- 所有权记录：链上记录 + 链下索引。
- 交易事件同步：mint/list/sale/transfer/cancel。
- 钱包绑定：平台账号与钱包地址绑定/解绑。

技术策略（建议）：

- 先定义链无关接口（`AssetService`），再做具体链适配器。
- 交易以事件回调驱动业务状态更新，避免强耦合同步调用。

## 6. 关键流程（按你强调的规则落地）

### 6.1 员工引入流程（必须）

1. 办公室管理员发起“引入员工”。
2. Office 模块调用 Employee 模块的 `assign` 接口（仅传员工ID与办公室ID）。
3. Employee 模块校验：
   - 员工存在且状态可分配；
   - 发起方有该办公室管理权限；
   - 若为电子员工，校验当前资产状态（未锁定/未交易中）。
4. 校验通过后写入 Assignment 关系。
5. 触发事件：`EmployeeAssignedToOffice`。
6. Office 侧刷新可视化座位/在线名单。

### 6.2 办公室切换流程（公司换办公室）

1. 公司购买或选择新办公室。
2. Office 实例切换，旧实例进入归档或出售状态。
3. Assignment 关系迁移策略执行（批量迁移或待确认迁移）。
4. 员工实体不变，员工资产不变。

### 6.3 电子员工交易（类比“跳槽”）

1. 持有方上架电子员工资产。
2. 成交后资产所有权变更。
3. 系统根据策略处理 Assignment：
   - 自动解除原办公室入驻，或
   - 进入“待新公司接收”状态。
4. 触发审计事件与通知。

## 7. 数据模型（逻辑层）

建议核心实体：

- `OfficeTemplate`：模板定义。
- `OfficeInstance`：公司实际使用的办公室。
- `EmployeeProfile`：员工主档。
- `EmployeeType`：HUMAN / DIGITAL。
- `EmployeeAssignment`：员工与办公室关系（可历史化）。
- `AssetToken`：资产主记录（类型、tokenId、链信息）。
- `AssetOwnership`：当前及历史所有权。
- `WalletBinding`：平台账号与钱包地址映射。

## 8. 权限与合规（MVP 基线）

- 角色：平台管理员、公司管理员、普通成员。
- 高风险动作（交易、转移、解绑）要求二次确认。
- 真实员工数据与电子员工交易数据分域存储。
- 所有关键操作记录审计日志（操作者、时间、来源、结果）。

## 10. 验收标准（第一版）

1. 可以创建至少两种办公室模板并实例化。
2. 办公室可引入/移出员工，但不能创建员工。
3. 员工离开某办公室后仍保留员工主体与资产状态。
4. 电子员工与办公室资产均可完成一次模拟交易。
5. 交易或迁移后，审计日志可追溯。
